# ccr-config-reading

## Purpose

Разобрать конфигурационный файл CCR, извлечь записи провайдеров и дедуплицировать их так, чтобы две записи, указывающие на один и тот же бэкенд с одним и тем же ключом, считались одной квотной корзиной.

## Requirements

### Requirement: Расположение конфига CCR

Система MUST читать конфиг из `~/.claude-code-router/config.json` по умолчанию. Путь MUST переопределяться флагом `-config`.

#### Scenario: Путь по умолчанию

- **WHEN** пользователь запускает `ccr-models-usage` без аргументов
- **THEN** система читает `~/.claude-code-router/config.json` (после раскрытия `~`)

#### Scenario: Переопределение пути

- **WHEN** пользователь запускает `ccr-models-usage -config /tmp/my-ccr.json`
- **THEN** система читает `/tmp/my-ccr.json` и игнорирует путь по умолчанию

#### Scenario: Конфиг отсутствует

- **WHEN** файл конфига не существует
- **THEN** система печатает однострочное сообщение об ошибке в stderr и завершается с ненулевым кодом

### Requirement: Извлечение провайдеров

Система MUST извлечь массив `Providers` из конфига и для каждой записи прочитать как минимум `name`, `api_base_url`, `api_key`, `models`. Неизвестные поля в записи провайдера MUST игнорироваться.

#### Scenario: Корректный конфиг

- **WHEN** в конфиге массив `Providers` из N записей
- **THEN** система создаёт N промежуточных структур `RawProvider`

#### Scenario: Пустой массив Providers

- **WHEN** в конфиге пустой массив `Providers`
- **THEN** система ничего не печатает и завершается с кодом 0

#### Scenario: Некорректный JSON

- **WHEN** файл конфига — не валидный JSON
- **THEN** система печатает однострочное сообщение об ошибке в stderr и завершается ненулевым кодом

### Requirement: Дедуп провайдеров

Система MUST схлопывать несколько записей, разделяющих пару `(hostname, api_key)`, в одну структуру `Provider`. Первое вхождение побеждает по полю `name`; список `models` — объединение всех подходящих записей в исходном порядке, с дедупликацией.

Hostname MUST быть компонентом hostname в нижнем регионе, извлечённым из `api_base_url` через `net/url`, с отбрасыванием порта, userinfo, пути, query и фрагмента. api_key сравнивается как сырая строка.

#### Scenario: Один хост, один ключ, разные пути

- **WHEN** две записи имеют `api_base_url`, различающиеся только путём (например, `https://opencode.ai/zen/go/v1/chat/completions` и `https://opencode.ai/zen/go/v1/messages`), и одинаковый `api_key`
- **THEN** система эмитит один `Provider` с `Host: "opencode.ai"`, объединённым списком `models` и `name` из первого вхождения

#### Scenario: Один хост, разные ключи

- **WHEN** две записи разделяют hostname, но имеют разные `api_key`
- **THEN** система эмитит две отдельные структуры `Provider`

#### Scenario: Разные хосты, один ключ

- **WHEN** две записи имеют разные hostname и одинаковый `api_key` (например, `zai` и `opencode` с одним ключом)
- **THEN** система эмитит две отдельные структуры `Provider`

#### Scenario: api_base_url некорректен

- **WHEN** `api_base_url` записи не парсится как URL
- **THEN** система пропускает эту запись и печатает однострочное предупреждение в stderr; она MUST NOT прерывать весь прогон
