# tdd-process

## Purpose

Зафиксировать дисциплину разработки через тесты (TDD) как обязательную методологию проекта: каждое наблюдаемое поведение сначала описывается падающим тестом, и только потом появляется имплементация, ровно настолько, чтобы тест позеленел. Линтер `golangci-lint` с `linters: enable: all` — must-pass gate.

## ADDED Requirements

### Requirement: Test-first

Каждая нетривиальная единица работы (новая публичная функция, новое наблюдаемое поведение, новый edge-case) MUST начинаться с написания падающего теста в `internal/<pkg>/<pkg>_test.go` (или `*_integration_test.go` под build-тегом). Тест MUST компилироваться и запускаться (`go test ./...`) — он красный по причине, которая ровно описывает целевое поведение. Имплементация пишется только после того, как красный тест зафиксирован.

#### Scenario: Новая функция в реестре

- **WHEN** разработчик добавляет новый публичный метод `Registry.X`
- **THEN** в репозитории появляется commit с падающим тестом на `Registry.X` ДО commit'а с реализацией
- **AND** история `git log -- internal/provider/` показывает, что файл `*_test.go` с этим тестом имеет более раннюю дату, чем соответствующий код

#### Scenario: Правка существующего поведения

- **WHEN** разработчик меняет контракт `Dedup` (например, формат ключа)
- **THEN** в diff сначала идёт изменение теста, отражающее новое поведение; commit с тестом красный, commit с реализацией — зелёный

### Requirement: Минимальная имплементация

Имплементация MUST покрывать ровно тот сценарий, который описывает тест. Никакого preemptive-рефакторинга, «заодно»-правок, добавления «полезных» хелперов или расширения публичной поверхности, не покрытого тестом.

#### Scenario: Лишний код

- **WHEN** тест на `Dedup` описывает только схлопывание по (host, key)
- **THEN** в реализации `Dedup` нет ни сортировки, ни валидации URL, ни других операций, не покрытых тестом
- **AND** последующие фичи добавляются отдельными red→green циклами

### Requirement: Refactor после зелёного

После того как целевой тест стал зелёным, разработчик MUST иметь возможность провести рефакторинг (выделение хелперов, переименование, устранение дублирования) без изменения ассертов существующих тестов. Изменения самих тестов вместе с кодом (rename, перенос) MUST трактоваться как новый red→green цикл — это новый «красный», а не silent-правка. Никакие рефакторинг-изменения не MUST ломать ранее зелёные тесты.

#### Scenario: Выделение хелпера

- **WHEN** `providers/zai` имеет две функции с дублирующим HTTP-кодом
- **THEN** разработчик выделяет общий `doJSON` после того, как оба существующих теста зелёные
- **AND** после рефакторинга `go test ./...` остаётся зелёным без правок ассертов

### Requirement: Линт как gate

Каждый PR MUST проходить `make lint` (т.е. `golangci-lint run ./...`) без ошибок и warning'ов. Конфиг — `.golangci.yml` с `linters: enable: all` плюс разумные `issues.exclusions` для `*_test.go` и сгенерированных моков.

#### Scenario: Линтер находит проблему

- **WHEN** код в `internal/render/` содержит `fmt.Errorf` с неконстантным форматом
- **THEN** `make lint` падает с указанием на этот файл и строку
- **AND** PR не может быть смёржен до исправления

#### Scenario: Тестовые файлы исключены из стилистических правил

- **WHEN** в `internal/provider/registry_test.go` используется `t.Errorf` с длинным сообщением
- **THEN** это НЕ считается линтер-ошибкой благодаря `exclusions` в `.golangci.yml`
- **AND** та же конструкция в `internal/render/text.go` — считается

### Requirement: PR состоит из red+green

Каждый PR (или каждый commit в линейной истории) MUST содержать и тест, и реализацию, делающую его зелёным. PR, который приносит только реализацию без теста, MUST быть отклонён; PR, который приносит только тест без прогресса по зелёному в последующих commit'ах того же PR, MUST быть отклонён, если это не первичный red-commit в red→green цикле.

#### Scenario: Одобрение PR

- **WHEN** PR содержит commit-1 с падающим тестом и commit-2 с реализацией, и оба проходят CI
- **THEN** PR одобряется

#### Scenario: Отклонение PR «только код»

- **WHEN** PR добавляет новую публичную функцию без сопутствующего теста
- **THEN** ревьюер просит разбить на red+green или отклоняет PR
